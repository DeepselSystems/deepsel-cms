# Phase 2b: Generate React Code with vibe_figma

## Goal
Use vibe_figma to generate pixel-accurate React + Tailwind components from the Figma design. This generated code becomes the **visual accuracy reference** for all Phase 4 conversion work.

## Prerequisites Check

Before starting, verify vibe_figma is working:

```bash
# Check vibefigma is available
npx vibefigma --help 2>/dev/null
if [ $? -ne 0 ]; then
  echo "ERROR: vibefigma not available. Install with: npm install -g vibefigma"
  exit 1
fi

# Check FIGMA_TOKEN is set
if [ -z "$FIGMA_TOKEN" ]; then
  echo "ERROR: FIGMA_TOKEN not set."
  echo "Get your token from: https://www.figma.com/settings → Personal Access Tokens"
  echo "Then: export FIGMA_TOKEN='your-token'"
  exit 1
fi

# Quick connectivity test (optional — use a known frame URL)
echo "Prerequisites OK: vibefigma available, FIGMA_TOKEN set"
```

If any prerequisite fails, stop and ask the user to fix it before continuing.

## Input

- `$MIGRATION_DIR/design-map.md` — lists all Category A pages with their Figma node URLs
- `$MIGRATION_DIR/design-data/frames.json` — frame metadata with `figmaUrl` field

## Process

### Step 1: Identify pages to generate

From `design-map.md`, collect all Category A pages. Each should have:
- Page name / slug
- Figma node URL (from frames.json or design-map.md)

Example:
```
Pages to generate:
1. homepage → https://www.figma.com/design/abc123/MyDesign?node-id=0%3A1
2. about → https://www.figma.com/design/abc123/MyDesign?node-id=10%3A25
3. contact → https://www.figma.com/design/abc123/MyDesign?node-id=20%3A50
```

### Step 2: Create output directories

```bash
mkdir -p $MIGRATION_DIR/vibe-output
```

### Step 3: Run vibe_figma for EACH page (SEQUENTIAL with delays)

For each Category A page:

```bash
npx vibefigma "<figma-node-url>" \
  --token $FIGMA_TOKEN \
  -c $MIGRATION_DIR/vibe-output/<page-slug>/ \
  -a $MIGRATION_DIR/vibe-output/<page-slug>/assets \
  --force
```

**Rate limit handling — NEVER skip a page:**
- **Wait 5 seconds between each page** — vibe_figma makes multiple Figma API calls internally
- If vibe_figma fails (exit code ≠ 0, or output contains "rate limit" / "429" / "too many requests"):
  1. Log: `"vibe_figma failed on <page>. Waiting 60 seconds..."`
  2. Wait **60 seconds**, retry
  3. If still failing, wait **120 seconds**, retry
  4. If still failing, wait **180 seconds**, retry
  5. If still failing after 4 attempts: **STOP and notify the user**:
     ```
     ⚠️ vibe_figma rate limited on page "<slug>" after 4 retries.
     This is the Figma API rate limit — it resets after a few minutes.
     Please wait 3-5 minutes, then confirm when ready to continue.

     Progress: X/Y pages completed.
     Remaining pages: <list remaining slugs>
     ```
  6. **Do NOT skip the page. Do NOT move to the next page.** Wait for user confirmation, then retry that exact same page.
- Track progress: `"Generated 2/5: about page ✓"`

**IMPORTANT: Run pages ONE AT A TIME.** Do NOT parallelize — vibe_figma itself makes multiple API calls per page.
**IMPORTANT: Every page must succeed.** The vibe_figma output is critical for visual accuracy in Phase 4. A missing page means Phase 4 has no reference for that page and will produce inaccurate code.

### Step 4: Verify output for each page

After each vibe_figma run, verify:
1. At least one `.tsx` file was generated in the output directory
2. Assets directory exists (may be empty if page has no images)
3. The generated code compiles (quick syntax check)

```bash
# List generated files
ls -la $MIGRATION_DIR/vibe-output/<page-slug>/

# Check for .tsx files
find $MIGRATION_DIR/vibe-output/<page-slug>/ -name "*.tsx" | head -5
```

If a page generated no output, log it as failed and note in the report.

### Step 5: Review generated code quality

For each generated page, quickly read the `.tsx` file to verify:
- It contains actual React/JSX code (not error messages)
- Tailwind classes are present
- Layout structure looks reasonable
- Images/assets are referenced

**Do NOT modify the vibe_figma output at this stage.** It's a reference — Phase 4 handles adaptation.

### Step 6: Collect and organize assets

After all pages are generated, gather assets:

```bash
# List all assets generated by vibe_figma
find $MIGRATION_DIR/vibe-output/ -path "*/assets/*" -type f
```

Create an asset summary:
```
Assets extracted by vibe_figma:
- homepage/assets/hero-bg.jpg (1440x600)
- homepage/assets/logo.svg
- homepage/assets/icon-1.svg
- about/assets/team-photo.jpg
- about/assets/office.jpg
Total: N assets across M pages
```

These assets will be copied to `themes/<theme-name>/assets/` in Phase 3.

### Step 7: Generate summary report

Create `$MIGRATION_DIR/vibe-output/SUMMARY.md`:

```markdown
# vibe_figma Code Generation Summary

## Pages Generated
| Page | Slug | Status | Files | Assets | Retries |
|------|------|--------|-------|--------|---------|
| Homepage | homepage | ✓ | 1 .tsx | 5 images | 0 |
| About | about | ✓ | 1 .tsx | 3 images | 2 (rate limit, waited 120s) |
| Contact | contact | ✓ | 1 .tsx | 2 images | 0 |

## Total
- Pages attempted: 3
- Pages succeeded: 3/3 ✓
- Total retries due to rate limits: 2
- Total assets: 10

## Notes
- (any observations about the generated code)
```

**All pages MUST show ✓.** If a page failed, you should have paused and asked the user to wait before continuing.

## What vibe_figma Produces

Typical output per page:
```
vibe-output/<page-slug>/
├── <ComponentName>.tsx    # React component with Tailwind CSS
└── assets/
    ├── image1.jpg         # Photos/backgrounds
    ├── icon1.svg          # Icons/illustrations
    └── logo.png           # Logos
```

The generated `.tsx` file typically contains:
- A single React component with the full page layout
- Exact Tailwind CSS classes matching Figma properties
- Image imports referencing the assets directory
- Hardcoded text content from the design
- Inline styles only where Tailwind has no equivalent

## What vibe_figma Does NOT Produce

The generated code is **static** — it needs CMS integration in Phase 4:
- No `WebsiteDataProvider` wrapper
- No dynamic menu (`Menu.tsx` with CMS data)
- No `ContentRenderer` for CMS-managed content
- No language switching (`useLanguage` hook)
- No interactive mobile menu (hamburger toggle)
- No blog templates (dynamic data)
- No Astro SSR routing (`.astro` files)
- No responsive variants (only generates from the single frame given)

**Think of vibe_figma output as a pixel-perfect screenshot converted to code.** Phase 4 brings it to life with CMS integration.

## Rules
- **NEVER skip the prerequisite check** — failing silently wastes time
- **NEVER parallelize vibe_figma calls** — rate limits will cause failures
- **NEVER modify vibe_figma output** — it's a reference for Phase 4, not the final code
- **NEVER skip a page due to rate limits** — wait and retry, notify user if needed
- **ALWAYS verify output** — check that .tsx files were actually generated
- **ALWAYS collect assets** — they need to be copied to the theme in Phase 3
- **If a page fails after 4 retries, STOP and ask the user to wait** — do NOT continue to the next page or Phase 3 with incomplete data

## Output
- `$MIGRATION_DIR/vibe-output/<page-slug>/` — generated React code per page
- `$MIGRATION_DIR/vibe-output/<page-slug>/assets/` — extracted images per page
- `$MIGRATION_DIR/vibe-output/SUMMARY.md` — generation report
- All assets organized and ready for Phase 3 copy
